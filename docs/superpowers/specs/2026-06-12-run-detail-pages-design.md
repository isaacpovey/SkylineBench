**Date:** 2026-06-12
**Status:** Approved pending spec review

Two related changes to the marketing site (`website/`):

   cards to **Fable 5, Opus 4.8, Haiku 4.5, Sonnet 4.5** (in that order), and

## Data reality (what a finished run records)

From `benchmark/runs/<id>/run-record.json` (schema v3) and `score.json`:

- `baseline` / `final` snapshots: `flow_mean`, `active_vehicles_mean`,
  `population`, `congested_meters`, `congested_junctions`.
- `flow_samples.baseline` / `flow_samples.final`: 8 samples each.
- `actions[]`: every tool call in order — `{seq, tool, cost}`.
- `tally`: `num_changes`, `money_spent`.
- `score.json`: `score`, `health`, `meters_reduction`, `junction_reduction`,
  `flow_gain`, plus `norm` / `weighted` breakdowns.
- `transcript.md`: full agent reasoning + tool calls (source for the curated
  narrative; **not** rendered verbatim).
- `model.txt`: the model slug (e.g. `fable`).

There is **no** continuous per-tick metric timeline. All charts below are
backed by the data above — before/after, the two settling windows, and values
derived from the action log.

## The four charts (inline SVG, generated by the build script)

Laid out as a **2×2 grid of cards** (chosen layout A).

1. **Before → after.** Paired bars per metric (baseline vs final) for the five
   snapshot metrics. Each metric pair is scaled to its own max so the
   different magnitudes (flow ~57–71 vs population ~31k) stay readable;
   absolute values labelled on the bars. Direction-of-good noted per metric.
2. **Flow settling curves.** Two line series over sample index 1–8:
   `flow_samples.baseline` vs `flow_samples.final`. Shows the settle and the
   final lift.
3. **Cumulative spend.** Step/line of cumulative `cost` over the action
   sequence (x = action number, which is also the change count). Conveys both
   money and number-of-changes in one chart. End label shows totals
   (197 changes · $1.24M for Fable 5).
4. **Action-type breakdown.** Bars per tool — call count and total cost.
   For Fable 5: bulldoze 6 ($0), build_road 11 ($57.8k), upgrade_road 180
   ($1.18M). Reveals the agent leaned almost entirely on road upgrades.

## Architecture

A new **Rust broker subcommand** generates each page (consistent with the
existing `timelapse` subcommand; no new toolchain). The static site keeps its
zero-runtime-dependency shape — charts are inline SVG, no JS chart library.

### Inputs / outputs

```
skylinebench build-page --narrative website/runs-src/<slug>.toml
```

- Reads the narrative TOML (curated content + a `run_dir` pointer).
- Reads `<run_dir>/run-record.json` and `<run_dir>/score.json` for chart data
  and key numbers.
- Emits `website/runs/<slug>.html` (static HTML, links the shared
  `colors_and_type.css` + `styles.css` and a new `runs.css`).
- Copies `<run_dir>/timelapse.mp4` → `website/assets/runs/<slug>.mp4` if
  present, so the hero video and the index card share one asset.

Re-running the command regenerates the page deterministically.

### Narrative source format

Curated content is a **source artifact** checked into the repo, separate from
the generated `benchmark/runs/` outputs. One TOML file per run at
`website/runs-src/<slug>.toml`, parsed with serde:

```toml
slug = "fable-5"
model_name = "Claude Fable 5"
map = "gridlock-v1"
run_dir = "benchmark/runs/20260612-121219"
verdict = "One-line summary of how the run went."

[[beat]]
title = "Survey"
body = "What the agent observed first and how it framed the problem."

[[beat]]
title = "First cuts"
body = "..."
# beats are text-only (chosen option D — no per-phase images)
```

The build script renders `[[beat]]` entries in order as the narrative
timeline. Key numbers, charts, score, and the timelapse all come from
`run_dir` automatically — the TOML only carries the human story and metadata.

### Page structure (top → bottom)

1. **Nav** — same as landing, with a "← Back to results" link to
   `index.html#results`.
2. **Hero** — model name + map, big composite score (`0.63 / 1.00`), the
   one-line verdict, and the autoplaying timelapse video.
3. **Key-numbers strip** — stat chips: flow 57→71, congested metres −64%,
   junctions 35→12, population 31.6k→31.2k, 197 changes, $1.24M.
4. **Charts** — the 2×2 grid above.
5. **Narrative timeline** — the curated `[[beat]]` play-by-play, text only.
6. **Footer** — same as landing.

### Styling

New `website/runs.css` holds detail-page-specific styles (hero, stat chips,
chart cards, timeline). Reuses existing CSS variables/typography from
`colors_and_type.css` and shared components from `styles.css`. The card→link
and "scored" pill styles are added to the shared `styles.css` since they
affect the landing page.

## Testing

- **Rust unit test** for `build-page`: given a fixture `run-record.json` +
  `score.json` + narrative TOML, the generated HTML contains the expected
  score, key numbers, chart data points (e.g. the 8 flow samples), and one
  section per beat. Asserts the SVG charts embed the right values rather than
  pixel-checking.
- **Manual check**: open `website/runs/fable-5.html` and `index.html` in a
  browser; confirm charts render, the timelapse plays, the card links through,
  and pending cards remain non-clickable.

## Out of scope

- Generating pages for the three pending models (no finished runs yet).
- A continuous per-tick metric timeline (not recorded).
- Any build pipeline beyond the single `build-page` subcommand (no bundler,
  no CI wiring).
